## 2. 핵심 룰 5개

### 2.1 기본 거부 (Deny by Default)

- 명시적으로 허용된 요청만 접근 가능
- 정의되지 않은 모든 요청은 차단

```text
if (no explicit permission) -> DENY
```

### 2.2 서버 측 권한 검사 (Server-side Enforcement)

- 모든 요청은 서버에서 권한 검사 수행
- 프론트엔드 제어는 보안 통제로 인정하지 않음

```text
on every request:
  validate(session.user, role, permission)
  if invalid -> DENY
```

### 2.3 객체 기반 접근 통제 (Object-level Access Control)

- 역할뿐 아니라 리소스 소유자/범위까지 검증
- 본인 데이터만 접근 가능하도록 제한

```text
if (resource.owner_id != session.user_id) -> DENY
or
if (resource not in user.scope) -> DENY
```

### 2.4 생성 / 승인 분리 (Separation of Duties)

- 생성, 검토, 승인 권한 분리
- 동일 사용자의 자기 승인 금지

```text
if (creator_id == approver_id) -> DENY
```

### 2.5 관리자 권한 분리 (Privileged Access Separation)

- 관리자 기능은 별도 계정에서만 수행
- 일반 계정에서 관리자 기능 접근 금지

```text
if (admin action && role != ADMIN) -> DENY
```

## 3. 권한 분리 6개 항목 (초안)

### 3.1 비회원 (Guest)

접근 가능 기능:

- 공개 메인 페이지 조회
- 기업 뉴스/공지 조회
- 공개 자료실 조회 및 다운로드
- 채용 공고 조회
- 회원가입
- 로그인
- 아이디/비밀번호 찾기
- 고객 문의 등록

접근 금지 기능:

- 마이페이지
- 지원 내역 조회
- 문의 답변 확인(본인 인증 없는 경우)
- 내부 그룹웨어 전체

### 3.2 외부 일반회원 (User)

접근 가능 기능:

- Guest 기능 전부
- 마이페이지 조회/수정
- 본인 문의 내역 조회
- 브로셔 다운로드
- 본인 계정 정보 수정

접근 금지 기능:

- 채용 지원자 전용 정보 열람
- 타인 문의/타인 정보 조회
- 내부 직원 기능
- 관리자 기능

### 3.3 지원자 (Applicant)

접근 가능 기능:

- User 권한 전부
- 채용 지원서 제출
- 이력서/자기소개서 업로드
- 본인 지원 내역 조회
- 본인 지원 상태 확인
- 본인 제출 문서 수정/취소(허용 시점까지만)

접근 금지 기능:

- 다른 지원자 정보 조회
- 평가 코멘트 확인
- 내부 검토 정보 조회
- 직원/관리자 화면 접근

### 3.4 내부 직원 (Employee)

접근 가능 기능:

- 임직원 전용 로그인
- 사내 공지 조회
- 사원 디렉토리 조회
- 전자결재 기안/조회
- 업무 게시판 사용
- 본인 부서 범위 내 문서 조회

접근 금지 기능:

- 고객 문의 일괄 처리
- 지원자 평가/합불 처리
- 권한 관리
- 시스템 설정 변경

### 3.5 업무 담당자/검토자 (Operator or Reviewer)

접근 가능 기능:

- Employee 권한 전부
- 고객 문의 열람/답변
- 지원자 서류 열람
- 지원 상태 변경
- 검토 메모 작성
- 특정 게시판/업무 문서 관리
- 첨부파일 검토

접근 금지 기능:

- 사용자 계정 삭제
- 역할 변경
- 시스템 설정 수정
- 전체 감사로그 삭제/변경
- 관리자 기능 전체

### 3.6 시스템 관리자 (Admin)

접근 가능 기능:

- 관리자 페이지 접근
- 사용자/직원 계정 생성·비활성화
- 역할 부여/회수
- 게시판/공지 관리
- 파일 승인 정책 관리
- 시스템 설정 변경
- 감사 로그 조회
- 운영 정책 변경

접근 금지/제한 기능:

- 일상 업무 처리 계정으로 겸용 금지
- 일반 브라우징/일반 업무용으로 사용 금지
- 필요 시 별도 관리자 로그인 경로/MFA 적용

## 4. 미확정 항목

- `Applicant`를 독립 Role로 둘지, `User + applicant_state` 모델로 둘지
- `Operator/Reviewer`를 `MANAGER`와 동일 계층으로 둘지, 하위 권한으로 분리할지
- 고객 문의 등록 권한을 Guest까지 허용할지(스팸/남용 리스크) 최종 확정 필요
- 관리자 계정의 MFA 강제 여부 및 예외 정책
