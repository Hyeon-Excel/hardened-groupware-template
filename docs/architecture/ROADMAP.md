# ROADMAP.md

# hardened-groupware-template 실행 로드맵

## 1. 문서 목적

본 문서는 `hardened-groupware-template` 프로젝트의 실제 수행 계획을 정의한다.  
기획안이 프로젝트의 목표와 구조를 설명하는 문서라면, 본 로드맵은 **누가, 어떤 단계에서, 무엇을, 어떤 순서로 수행할지**를 정리한 실행 계획 문서이다.

본 프로젝트는 다음 흐름으로 진행한다.

1. 인프라 및 서비스 구축
2. 보안 검증 기준 정리
3. 취약점 분석 및 침투 시나리오 수행
4. 보완 및 하드닝
5. 재검증 및 보고서 작성
6. 공개 배포 준비

---

## 2. 프로젝트 수행 원칙

### 2.1 핵심 원칙

- 기능 구현보다 **구조적 완성도와 재현성**을 우선한다.
- 취약점은 발견으로 끝내지 않고 **보완과 재검증**까지 수행한다.
- 공개 웹 서비스와 내부 그룹웨어를 **분리된 서비스 경계**로 유지한다.
- 공개 데이터와 내부 데이터를 분리하고, 파일은 검역 후 사용한다.
- 제출용 문서와 내부 운영 문서를 구분하여 관리한다.

### 2.2 보안 실험 원칙

- 사용한 테스트 악성코드(멀웨어 등)은 배포하지 않는다.
- 멀웨어 평가는 **무해한 시뮬레이션 페이로드, 검역 평가용 샘플, 탐지/차단 정책 검증용 테스트 파일** 중심으로 수행한다.
- 외부 업로드 파일은 반드시 검역 저장소를 거친다.
- 침투 및 공격 시나리오는 승인된 폐쇄형 실습 환경에서만 수행한다.

## 3. 인력 구성 및 역할 분담 (R&R)

본 프로젝트는 총 **5명**이 수행한다.

- 서비스 개발 담당: 2명 (최현수, 이성준)
- 인프라 구축 담당: 1명 (김가영)
- 침투 시나리오 / 테스트용 시뮬레이션 페이로드 담당: 2명 (이지원, 심수아)

단, **취약점 분석, 시스템 개선, 인프라 개선, 최종 검토**는 전 인원이 함께 수행한다.

### 3.1 서비스 개발 담당 (외부 서비스) - 최현수

담당 범위:

- 공개 웹 프론트엔드
- 공개 웹 백엔드
- 일반 사용자 인증 및 세션 처리
- 뉴스, 자료실, 고객센터, 채용 기능
- 공개 데이터베이스 연동
- 공개 서비스 취약점 수정

### 3.2 서비스 개발 담당 (내부 서비스) - 이성준

담당 범위:

- 내부 그룹웨어 프론트엔드
- 내부 그룹웨어 백엔드
- 임직원 인증 / RBAC
- 전자결재 / 사원 디렉토리 / 관리자 기능
- 내부 데이터베이스 연동
- 내부 서비스 취약점 수정

### 3.3 인프라 구축 담당 - 김가영

담당 범위:

- VMware 3-Tier 실습 환경
- Nginx 프록시 구성
- 네트워크 / 방화벽 / 접근 정책
- Docker Compose 실행 환경
- 파일 저장소 / 검역 흐름 배치
- 시스템 실행 및 배포 구조 정리

### 3.4 침투 시나리오 / 테스트 페이로드 담당 - 이지원, 심수아

담당 범위:

- 인증 / 인가 우회 시나리오 설계
- 공개 서비스와 내부 서비스 경계 검증
- 대표 공격 흐름 재현
- 취약점 재현 절차 문서화
- 파일 검역 평가용 샘플 설계
- 무해한 시뮬레이션 파일 및 테스트 페이로드 준비
- 차단 / 격리 / 검토 흐름 검증
- 탐지 및 방어 결과 정리

### 3.5 전 인원 공통 책임

- ASVS 체크리스트 점검
- 취약점 분석
- 시스템 및 인프라 개선
- 하드닝 전후 비교
- 최종 보고서 작성
- 발표 및 배포 준비

## 5. Phase 기반 마일스톤

### 5.1 예상 일정 (기준선)

아래 일정은 **예상 기간 기준선**이며, 실제 시작일은 팀 킥오프일에 맞춰 조정한다.

기준선 예시:

- 킥오프: 2026-04-20
- 목표 종료: 2026-05-06 (약 17일)

| Phase   | 예상 기간 | 기준선 예시 기간 (2026년) | 산출물/게이트              |
| ------- | --------- | ------------------------- | -------------------------- |
| Phase 1 | --        | 4/14 ~ 4/16               | 아키텍처/API/DB 초안 확정  |
| Phase 2 | 3일       | 4/17 ~ 4/19               | 외부/내부 MVP 1차 동작     |
| Phase 3 | 3일       | 4/19 ~ 4/22               | 3-Tier/파일 처리 구조 동작 |
| Phase 4 | 3일       | 4/22 ~ 4/24               | 1차 취약점 분석 결과       |
| Phase 5 | 3일       | 4/25 ~ 4/28               | 보완 + 검역/방어 평가      |
| Phase 6 | 3일       | 5/4 ~ 5/6                 | 재검증 + 최종 공개 준비    |

### 5.2 Phase 의존성과 병렬 수행 원칙

핵심 의존성:

- Phase 2는 Phase 1의 API/DB 초안 고정을 전제로 시작한다.
- Phase 3은 Phase 2 산출물과 병행 가능하나, 최종 통합 검증은 Phase 2 완료가 필요하다.
- Phase 4는 Phase 2/3의 실행 가능한 통합 환경을 전제로 한다.
- Phase 5는 Phase 4 취약점 백로그 확정 이후 본격 착수한다.
- Phase 6는 Phase 5 수정 사항이 반영된 브랜치 기준으로 수행한다.

병렬 수행 가이드:

- 외부 서비스 팀과 내부 서비스 팀은 Phase 2에서 병렬 개발한다.
- 인프라 팀은 Phase 2 중반부터 Phase 3 선행 작업(네트워크, 저장소, 프록시)을 병렬 진행한다.
- 침투/검역 평가 팀은 Phase 2 후반부터 테스트 시나리오와 페이로드를 선행 준비한다.
