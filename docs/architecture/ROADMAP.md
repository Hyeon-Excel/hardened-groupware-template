# ROADMAP.md

# hardened-groupware-template 실행 로드맵

## 1. 문서 목적

본 문서는 프로젝트의 실제 수행 계획(누가, 언제, 무엇을, 어떤 순서로 수행할지)을 정의한다.
`PLANNING.md`의 방향을 일정/역할/산출물 단위로 실행 가능하게 정리한 문서다.

---

## 2. 프로젝트 수행 원칙

### 2.1 핵심 원칙

- 기능 구현과 보안 분석을 분리하지 않고 v0→분석→v1로 연속 수행한다.
- 취약점은 발견으로 끝내지 않고 보완·재검증까지 완료한다.
- external/internal 서비스 경계를 유지한다.
- DB는 2개로 고정하고 파일은 단일 오브젝트 스토리지 + 상태 머신으로 관리한다.

### 2.2 보안 실험 원칙

- 모의해킹/침투 테스트는 승인된 폐쇄형 환경에서만 수행한다.
- 외부 서비스·타사 자산 대상 공격은 범위에서 제외한다.
- 테스트 페이로드는 무해한 샘플 또는 승인된 실습 파일만 사용한다.

---

## 3. 인력 구성 및 역할 분담 (R&R)

총 5명 체계:

- 외부 서비스 개발: 1명
- 내부 서비스 개발: 1명
- 인프라/배포: 1명
- 침투 시나리오/테스트 페이로드: 2명

공통 책임:

- ASVS 점검
- 취약점 분석/백로그 관리
- 하드닝 및 재검증
- 최종 보고서 작성/발표 준비

---

## 4. WBS(작업 분류)

### 4.1 기획/정합성 고정

- PLANNING/ARCHITECTURE/API/DB 문서 정합성 확정
- v0/v1 경계(무엇을 v0에서 제외할지) 고정

### 4.2 v0 MVP 구현

- external 우선 구현
- internal은 최소 구조(로그인 + 기본 페이지) 확보
- 세션/CSRF/기본 입력 검증 적용
- 파일은 `uploaded_file` 상태 기록(`V0_SCAN_DISABLED`)만 적용

### 4.3 분석 페이즈

- 시나리오 기반 모의해킹(Burp/ZAP + 수동)
- ASVS 매핑
- CodeQL/Dependency Review
- finding backlog 작성(open/risk/priority)

### 4.4 v1 하드닝

- finding 기반 보완 패치
- 파일 보안 카드(`low/medium/high/high+alpha`) 중 선택 적용
- fail-closed, 동시성 락, 재동기화 잡 반영

### 4.5 재검증/문서화

- 동일 시나리오 재실행
- 전후 비교표 작성
- 보고서/가이드/공개본 정리

## 5.2 Phase 의존성

- Phase 1은 Phase 0 문서 고정 이후 시작
- Phase 2는 v0 freeze 커밋 기준 수행
- Phase 3은 finding backlog 승인 이후 착수
- Phase 4는 Phase 3 패치 반영 브랜치 기준
- Phase 5는 재검증 결과 확정 이후 진행

## 5.3 병렬 수행 가이드

- 외부/내부 개발자는 Phase 1에서 병렬 가능
- 인프라 담당은 Phase 1 후반부터 배포/재현 환경 병행 준비
- 분석 담당은 Phase 1 후반부터 시나리오/페이로드 사전 준비

## 6. Phase 상세

### Phase 0 — 문서 기준선 확정

목표:

- 정본 문서 간 충돌 제거
- API 계약과 DB 스키마 기준 확정

완료 조건:

- `PLANNING`, `ARCHITECTURE`, `API`, `openapi.yaml`, `database/*` 정합성 확보

### Phase 1 — v0 MVP 구축

목표:

- external 핵심 흐름 구현
- internal 최소 골격 확보

주요 작업:

- 로그인/세션/CSRF
- 뉴스/공지/자료실/문의/지원 기본 플로우
- `uploaded_file` 상태 기록(v0 기본값)
- internal은 로그인/기본 페이지 중심 최소 골격만 유지(external 연동 없음)

완료 조건:

- 사용자 핵심 플로우 동작
- v0 freeze 커밋 SHA 확정

### Phase 2 — 모의해킹 및 취약점 분석

목표:

- v0 공격면 검증
- 취약점 재현 가능한 형태로 기록

주요 작업:

- BOLA/IDOR
- 세션/CSRF
- 파일 업로드/다운로드(Stored/second-order 포함)
- external/internal 경계 검증을 위한 최소 연동 활성화
- 서비스 경계 검증
- CodeQL/Dependency Review 정리

완료 조건:

- 취약점 백로그 확정
- finding ID 체계 확정

### Phase 3 — v1 하드닝

목표:

- finding 기반 보완
- 보안 카드 적용

주요 작업:

- 인증/인가 강화
- 파일 상태 게이트/스캔 카드 적용
- fail-closed, 락, 재동기화 적용
- 감사/운영 로그 보강

완료 조건:

- 주요 finding closed 또는 risk-accepted 처리

### Phase 4 — 재검증

목표:

- 동일 시나리오 재실행
- 회귀 여부 검증

완료 조건:

- 전후 비교표 확정
- 미해결 항목에 대한 잔여 리스크 기록

### Phase 5 — 최종 문서화/공개 준비

목표:

- 제출/공개 가능한 형태로 정리

주요 작업:

- 최종 보고서
- 실행/재현 가이드
- 민감정보 제거
- Docker Compose 재현 점검

완료 조건:

- 최종 산출물 리뷰 완료
- 공개 브랜치/태그 준비 완료
