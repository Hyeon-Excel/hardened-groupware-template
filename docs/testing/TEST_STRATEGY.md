# TEST_STRATEGY.md

## 1. 목적

이 문서는 구현 단계에서 테스트 범위와 우선순위를 일관되게 유지하기 위한 전략 기준이다.

적용 범위:

- `external-web`, `internal-web`
- `external-api`, `internal-api`
- `file-worker`

---

## 2. 테스트 피라미드 기준

프로젝트 기본 비율:

- 단위 테스트(Unit): 60%
- 통합 테스트(Integration): 30%
- E2E 테스트: 10%

원칙:

- 비즈니스 규칙은 단위 테스트로 빠르게 검증한다.
- 인증/인가/DB/외부 연계는 통합 테스트로 검증한다.
- 핵심 사용자 플로우만 E2E로 최소 집합 유지한다.

---

## 3. 백엔드 테스트 전략

대상:

- `external-api`
- `internal-api`
- `file-worker`

### 3.1 단위 테스트

- 서비스 계층 비즈니스 규칙
- 권한 판정 유틸/정책 로직
- 파일 정책 판정(확장자/MIME/매직바이트 규칙)

### 3.2 통합 테스트

- API 엔드포인트 + 인증/인가 필터 체인
- DB 리포지토리 쿼리
- `internal-api -> external-api` 브리지 호출 (mock/stub 또는 테스트 전용 endpoint)
- `file_scan_jobs` polling 및 상태 전이

### 3.3 보안 회귀 테스트(자동화 필수)

- CSRF 미포함 상태 변경 요청 차단
- 세션 없는 보호 API 접근 차단(401)
- 권한 부족 접근 차단(403)
- 서비스 토큰 누락/불일치 차단(401/403)
- IDOR 방어: 타 사용자 리소스 조회 시 404 반환

---

## 4. 프론트엔드 테스트 전략

대상:

- `external-web`
- `internal-web`

### 4.1 컴포넌트 테스트

- 폼 검증
- 상태 배지(`FileStatusBadge`, `ApprovalStatusBadge` 등)
- 에러/로딩/빈 상태 렌더링

### 4.2 페이지/플로우 E2E

최소 필수 플로우:

- 외부 로그인 → 마이페이지 접근
- 문의 등록 → 내 문의 목록/상태 확인
- 채용 지원서 제출 → 상태 확인
- 내부 로그인 → 결재 처리
- 관리자 검역 로그 조회

---

## 5. 커버리지 및 품질 게이트

- 백엔드 단위+통합 커버리지 목표: 70% 이상
- 보안 핵심 모듈(인증/인가/파일/브리지)은 80% 이상 권장
- PR 병합 전 최소 조건:
  - 관련 테스트 통과
  - 신규 보안 규칙 변경 시 보안 회귀 테스트 1개 이상 추가

---

## 6. CI 실행 기준

- 빠른 체크(PR): 단위 테스트 + 핵심 통합 테스트 + API 계약 가드
- 확장 체크(main/develop): 전체 통합 + E2E 스모크
- 취약점 보완 PR: 보안 회귀 테스트 포함 필수
